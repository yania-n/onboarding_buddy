# **3. The Cloud Operations Role — What You Do**

The Cloud Operations team is the reliability and operational excellence function of the Cloud Infrastructure & Platform Team. Three specialist roles work together as one squad:

## **Site Reliability Engineering (SRE)**

### **What SREs own**

You own the reliability and observability of Nexora's production cloud environment. This means defining and tracking SLOs (Service Level Objectives), building and maintaining monitoring and alerting infrastructure, leading incident response, and driving post-incident reviews that prevent recurrence.

- Define SLOs and SLIs for all critical platform services

- Own the observability stack (Datadog, PagerDuty) — dashboards, alerts, runbooks

- Lead on-call rotations and incident response for P1/P2 incidents

- Conduct blameless post-incident reviews (PIRs) within 48 hours of P1 incidents

- Drive reliability improvements via error budget tracking and toil reduction

## **FinOps Analysis**

### **What FinOps Analysts own**

You own cloud cost visibility and optimisation across all Nexora cloud accounts. Your job is to ensure Nexora gets maximum value from its cloud spend — tracking, allocating, forecasting, and reducing costs without compromising reliability.

- Maintain cloud cost dashboards by account, service, team, and environment

- Run monthly cost reviews with each engineering team and the CTO office

- Identify and action cost optimisation opportunities (rightsizing, Reserved Instances, Savings Plans)

- Forecast cloud spend for the annual budget and quarterly reforecasts

- Define and enforce tagging standards that enable accurate cost allocation

## **Cloud Security Engineering**

### **What Cloud Security Engineers own**

You own the security posture of Nexora's cloud environments — ensuring our infrastructure meets compliance requirements, passes external audits, and is configured to prevent breaches.

- Own the cloud security posture management (CSPM) tooling and remediation backlog

- Define and enforce cloud security baselines (IAM policies, network security groups, encryption standards)

- Lead on security-related changes to cloud infrastructure — review and approve before deployment

- Support external compliance audits (SOC 2, ISO 27001, GDPR) for cloud infrastructure scope

- Respond to cloud security incidents in partnership with the Cybersecurity team

## **What the Cloud Ops Team Does NOT Own**

- Application-layer security (owned by Cybersecurity & InfoSec team)

- Provisioning new cloud resources from scratch (Platform Engineering)

- Writing application code or product features

- CI/CD pipeline tooling (Developer Experience team)

# **4. How We Work — Sprints ****&**** On-Call**

## **2-Week Sprint Cadence**

All three Cloud Infrastructure & Platform sub-teams run on a shared 2-week sprint. The Cloud Ops backlog covers reliability improvements, security remediations, cost optimisation projects, observability enhancements, and internal tooling.

| **Sprint Planning** | Monday Week 1, 1.5h — whole team; commit to sprint backlog |
| --- | --- |
| **Daily Stand-up** | Every morning, 15 min — progress, blockers, any live incidents or alerts |
| **Infra Refinement** | Wednesday Week 1, 90 min — groom and estimate upcoming backlog items |
| **Sprint Review** | Friday Week 2, 45 min — demo completed work to IT Director and stakeholders |
| **Retrospective** | Friday Week 2, 45 min — team process and ways-of-working improvements |

## **On-Call Rotation**

SREs and Cloud Security Engineers participate in a 24/7 on-call rotation for production incidents. FinOps Analysts are not on the on-call rota. Key on-call principles:

- On-call rotation is weekly — one primary and one secondary engineer at all times

- Alerts are routed via PagerDuty — ensure your PagerDuty app is installed and notifications enabled on Day 1

- P1 (service down, data loss risk): respond within 5 minutes, engage secondary within 15 minutes

- P2 (degraded performance, partial outage): respond within 30 minutes

- On-call engineers do not take sprint work on-call weeks — capacity is 20% sprint work, 80% reactive

- Time-in-lieu is given for on-call weekends — log in Workday under 'On-Call Compensation'

|  | **ON-CALL ONBOARDING** New joiners are added to the on-call rotation only after completing 30 days on the team, shadowing 2 full on-call weeks, and being signed off by the Cloud Ops Lead. You will not be paged before you are ready. |
| --- | --- |

## **Change Management**

All infrastructure changes to the production environment follow Nexora's Change Advisory Board (CAB) process:

| **Standard change** | Pre-approved, low-risk changes (e.g. tagging updates, alert threshold tweaks) — no CAB needed, deploy via PR merge |
| --- | --- |
| **Normal change** | Moderate-risk changes — require CAB approval (submit Tuesday for Thursday CAB meeting) |
| **Emergency change** | P1 incident remediation — CAB Chair approval by phone/Slack; retrospective CAB review within 48h |

# **5. Infrastructure Request ****&**** Change Flow**

## **How Infrastructure Work Reaches Us**

| **Stage** | **Owner** | **What happens** |
| --- | --- | --- |
| **1. Request raised** | Engineering team / Business stakeholder | New infra requirement, security finding, cost concern, or reliability issue raised via Jira Service Desk |
| **2. Triage ****&**** scoping** | Cloud Ops Lead + relevant engineer | Request triaged in Cloud Ops Jira; categorised (reliability/security/cost/new infra); routed to right sub-team |
| **3. Design ****&**** review** | Platform Eng + Cloud Ops | Architecture design for significant changes; security review; cost impact assessed |
| **4. Change request submitted** | Requesting engineer | CAB change request submitted if required; IaC PR opened for peer review |
| **5. Deployment** | Platform Eng / Cloud Ops | Change deployed to non-prod, validated, then promoted to production via approved pipeline |
| **6. Monitoring ****&**** closure** | Cloud Ops SREs | Post-deployment monitoring for 24–48h; alerts validated; Jira ticket closed with deployment notes |

## **Self-Service vs Managed Infrastructure**

Not every request needs to come to Cloud Ops. Engineering teams can self-serve many common infrastructure needs via the Internal Developer Platform (owned by Developer Experience). The distinction:

| **Self-service (no ticket needed)** | Spinning up sandbox environments, deploying to existing accounts, standard EC2/RDS sizes — via the IDP |
| --- | --- |
| **Cloud Ops ticket required** | New AWS accounts, VPC changes, IAM policy changes, production database changes, security baseline exceptions, cost anomaly investigations |

# **7. Cloud Standards, Security ****&**** Cost Governance**

## **Cloud Standards**

- All infrastructure must be defined as code (Terraform) — no manual changes in production consoles

- IaC modules are stored in GitHub: nexora-org/infra-modules — PRs require 2 approvals

- Every AWS resource must have the following tags: team, environment, cost-centre, data-classification

- Encryption at rest and in transit is mandatory for all production resources — no exceptions

- No public S3 buckets — ever. Automated CSPM rules will alert and auto-remediate within 1 hour

## **Cloud Security Baseline**

| **IAM** | Least-privilege by default; no long-lived access keys; all human access via SSO; MFA mandatory |
| --- | --- |
| **Network** | VPC per environment; no 0.0.0.0/0 ingress rules; all traffic through WAF for public endpoints |
| **Data** | Encryption at rest (AES-256); in transit (TLS 1.2+); KMS key rotation enabled |
| **Logging** | CloudTrail enabled in all accounts; logs centralised in security account; 12-month retention |
| **Secrets** | No secrets in code or environment variables — use AWS Secrets Manager or HashiCorp Vault |

## **Cost Governance**

- Cloud budget is reviewed monthly — teams over budget by > 10% require a cost review with FinOps

- All production resources must have cost allocation tags before deployment — CI/CD will block untagged resources

- Reserved Instance and Savings Plan recommendations reviewed quarterly by FinOps and approved by CTO

- Development and staging environments must be scheduled off overnight and weekends (automated via Instance Scheduler)

|  | **COST ANOMALY ALERTS** If you see an unexpected cost spike in the #cloud-cost-alerts channel, investigate immediately. The FinOps team responds to anomalies > $500/day within 4 hours. Flag anything that doesn't look right — it's always better to investigate a false alarm than miss a real one. |

# **8. Incident Management**

## **Severity Levels**

| **P1 — Critical** | Full service outage or data loss risk. On-call SRE responds in 5 min. Incident bridge opened. Executive stakeholders notified within 30 min. |
| --- | --- |
| **P2 — Major** | Significant degradation affecting multiple users. On-call SRE responds in 30 min. Incident channel opened in Slack. |
| **P3 — Minor** | Partial degradation, workaround available. Respond next business day. Ticket created in Jira. |
| **P4 — Low** | Cosmetic or informational issue. Backlog item created; prioritised in next sprint. |

## **Incident Response Process**

- PagerDuty alert fires → On-call SRE acknowledges within SLA

- Open incident channel in Slack: #inc-[YYYYMMDD]-[short-description]

- Assign Incident Commander (IC) — on-call SRE for P2; senior SRE or Cloud Ops Lead for P1

- Diagnose and mitigate — runbooks in Confluence: Cloud Ops > Runbooks

- Communicate updates every 15 minutes (P1) or 30 minutes (P2) in #cloud-incidents

- Resolve and close PagerDuty alert

- Post-Incident Review within 48h — blameless, action items tracked in Jira

|  | **BLAMELESS CULTURE** We do not assign blame for incidents. Systems fail; people respond. Our PIR process asks 'what failed?' and 'how do we prevent it?' not 'who caused it?'. If you are ever made to feel responsible for a system failure, speak to the Cloud Ops Lead. |

# **9. Definition of Ready ****&**** Done**

## **Definition of Ready — an infrastructure task is Ready when:**

- The requirement is clearly stated: what problem does this solve, or what improvement does it deliver?

- The infrastructure scope is defined (which accounts, regions, services)

- Security and compliance implications have been assessed

- Cost impact has been estimated by FinOps

- Rollback plan is documented for any production change

- CAB ticket submitted if a normal or emergency change

## **Definition of Done — infrastructure work is Done when:**

- Change is deployed to production and validated against defined acceptance criteria

- Post-deployment monitoring confirmed stable for 24–48 hours

- IaC is merged to main and documentation updated in Confluence

- Cost tags are applied and verified in the billing dashboard

- Security scan passed (Wiz/Prisma Cloud findings resolved)

- Jira ticket updated to 'Done' with deployment notes and runbook links

# **12. Your First 30 Days**

## **Week 1 — Orient**

- Submit Day 1 access requests: AWS read-only, Datadog, PagerDuty, GitHub, Jira, Confluence

- Install PagerDuty mobile app and verify your notification settings — do this on Day 1

- Complete mandatory NexoraLearn modules: Nexora AWS Architecture Overview, Cloud Security Baseline, Data Privacy

- Meet your Cloud Ops Lead, squad members, and onboarding buddy in 1:1s

- Read the Cloud Platform Architecture Overview in Confluence: Cloud Infra > Architecture

- Join all required Slack channels and introduce yourself in #cloud-platform

## **Week 2 — Immerse**

- Shadow your buddy through the full on-call workflow — observe how alerts are triaged and incidents handled

- Explore the Datadog dashboards — understand what 'healthy' looks like for the platform

- Attend sprint ceremonies and your first Infra Refinement session

- Review the last 3 Post-Incident Reviews in Confluence to understand how we learn from incidents

- Submit AWS write access and Terraform Cloud access requests if not already done

## **Weeks 3–4 — Contribute**

- Complete 'Terraform at Nexora' training in NexoraLearn

- Take ownership of one sprint task (backlog improvement, alert tuning, cost tagging, or runbook update) with buddy support

- Shadow your first on-call week (observe only — your buddy is primary)

- Attend your first Cloud Cost Review or Security Posture Review (whichever falls in this period)

- Complete your 30-day check-in with your manager in Workday
